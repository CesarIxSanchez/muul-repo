3. Automatización (Triggers Activos)
1.	handle_new_user(): Se ejecuta en cada INSERT sobre auth.users. Extrae metadatos (Google/Facebook) y crea la fila en perfiles.
2.	check_for_badges(): Se ejecuta en cada INSERT sobre visitas. Cuenta el histórico y otorga medallas en usuario_insignias automáticamente.
3.	actualizar_calificacion_negocio(): Se ejecuta al insertar, modificar o borrar una fila en resenas. Recalcula instantáneamente la calificacion_promedio y el total_resenas en la tabla negocios.
4. Seguridad (Row Level Security - RLS)
•	Perfiles/Visitas/Recorridos: El turista solo puede leer o modificar su propia información (auth.uid() = usuario_id).
•	Negocios/Catálogos/POIs: Todo usuario puede aplicar SELECT (lectura pública). Solo el propietario o administrador puede hacer UPDATE/INSERT.
•	Reseñas: Públicas para lectura. Un turista solo puede crear/editar su propia reseña.

1. Sistema de Chat B2B (Business to Business)
Este bloque incluye la creación de tablas, índices de rendimiento para una carga rápida y las políticas de seguridad (RLS) para proteger la privacidad de las empresas:
-- 1. Tabla de Conversaciones
create table public.chats_b2b (
  id uuid primary key default gen_random_uuid(),
  propietario_1_id uuid references public.perfiles(id) on delete cascade,
  propietario_2_id uuid references public.perfiles(id) on delete cascade,
  creado_en timestamp with time zone default now(),
  unique(propietario_1_id, propietario_2_id) -- Evita salas duplicadas 
);

-- 2. Tabla de Mensajes
create table public.mensajes_b2b (
  id uuid primary key default gen_random_uuid(),
  chat_id uuid references public.chats_b2b(id) on delete cascade,
  remitente_id uuid references public.perfiles(id) on delete cascade,
  contenido text not null,
  leido boolean default false,
  creado_en timestamp with time zone default now()
);

-- 3. Índice para optimización de carga 
create index idx_mensajes_chat on public.mensajes_b2b(chat_id, creado_en);

-- 4. Seguridad (RLS) [cite: 118, 120]
alter table public.chats_b2b enable row level security;
alter table public.mensajes_b2b enable row level security;

-- Política: Solo los participantes ven sus chats [cite: 120]
create policy "Ver mis chats" on public.chats_b2b for select 
  using (auth.uid() = propietario_1_id or auth.uid() = propietario_2_id);

-- Política: Solo los participantes ven los mensajes de sus chats [cite: 120]
create policy "Ver mensajes de mis chats" on public.mensajes_b2b for select 
  using (
    chat_id in (
      select id from public.chats_b2b 
      where propietario_1_id = auth.uid() or propietario_2_id = auth.uid()
    )
  );

-- Política: Crear chats y enviar mensajes [cite: 120]
create policy "Crear chats" on public.chats_b2b for insert 
  with check (auth.uid() = propietario_1_id or auth.uid() = propietario_2_id);

create policy "Enviar mensajes" on public.mensajes_b2b for insert 
  with check (
    auth.uid() = remitente_id and
    chat_id in (
      select id from public.chats_b2b 
      where propietario_1_id = auth.uid() or propietario_2_id = auth.uid()
    )
  );

2. Módulo de Interacción Turista-Empresa
Para cerrar la arquitectura de comunicación, este módulo de Interacción Turista-Empresa es vital. Permitirá que los viajeros resuelvan dudas sobre productos , consulten horarios específicos o realicen consultas antes de visitar el local, aumentando la confianza y la conversión para las PyMEs.
Dominio	Nombre de la Tabla	Propósito Principal	Relaciones Clave
Comunicación	chats_soporte	Canal directo entre un turista y un negocio.	perfiles (N:1), negocios (N:1).
Comunicación	mensajes_soporte	Contenido de las dudas, respuestas y fotos enviadas.	chats_soporte (N:1), perfiles (N:1).

Diccionario de Datos Detallado
Dominio de Comunicación (Turista-Empresa)
chats_soporte
•	id (UUID, PK): Identificador único de la conversación.
•	usuario_id (UUID, FK): El turista que inicia la pregunta.
•	negocio_id (UUID, FK): El establecimiento que recibe la consulta.
•	creado_en (Timestamp): Fecha de inicio del contacto.
•	Nota: Restricción UNIQUE (usuario_id, negocio_id) para mantener un solo hilo histórico por cliente.
mensajes_soporte
•	id (UUID, PK): Identificador del mensaje.
•	chat_id (UUID, FK): Conversación a la que pertenece (ON DELETE CASCADE).
•	remitente_id (UUID, FK): ID de quien escribe (ya sea el turista o el dueño).
•	contenido (Text): El cuerpo de la pregunta o respuesta.
•	leido (Boolean): Estado para notificaciones en el frontend. Default: false.
•	creado_en (Timestamp): Hora exacta del mensaje.

Código SQL y Seguridad (RLS)
Este script crea las tablas y asegura que solo el cliente interesado y el dueño del negocio puedan leer los mensajes.
-- 1. Crear Tablas de Comunicación Turista-Empresa
create table public.chats_soporte (
  id uuid primary key default gen_random_uuid(),
  usuario_id uuid references public.perfiles(id) on delete cascade,
  negocio_id uuid references public.negocios(id) on delete cascade,
  creado_en timestamp with time zone default now(),
  unique(usuario_id, negocio_id)
);

create table public.mensajes_soporte (
  id uuid primary key default gen_random_uuid(),
  chat_id uuid references public.chats_soporte(id) on delete cascade,
  remitente_id uuid references public.perfiles(id) on delete cascade,
  contenido text not null,
  leido boolean default false,
  creado_en timestamp with time zone default now()
);

-- 2. Índices de Rendimiento
create index idx_mensajes_soporte_chat on public.mensajes_soporte(chat_id, creado_en);

-- 3. Seguridad (Row Level Security)
alter table public.chats_soporte enable row level security;
alter table public.mensajes_soporte enable row level security;

-- Política: El turista ve sus chats; el dueño ve los chats de su negocio
create policy "Ver mis chats de soporte" on public.chats_soporte for select 
  using (
    auth.uid() = usuario_id or 
    auth.uid() in (select propietario_id from public.negocios where id = negocio_id)
  );

-- Política: Acceso a los mensajes si eres parte del chat
create policy "Ver mensajes de soporte" on public.mensajes_soporte for select 
  using (
    chat_id in (
      select id from public.chats_soporte 
      where usuario_id = auth.uid() or 
      negocio_id in (select id from public.negocios where propietario_id = auth.uid())
    )
  );

-- Política: Insertar mensajes
create policy "Enviar mensajes de soporte" on public.mensajes_soporte for insert 
  with check (
    auth.uid() = remitente_id and
    chat_id in (
      select id from public.chats_soporte 
      where usuario_id = auth.uid() or 
      negocio_id in (select id from public.negocios where propietario_id = auth.uid())
    )
  );

Consideraciones para el Equipo
•	Privacidad Total: Gracias al RLS, ningún otro usuario puede interceptar las dudas o datos personales enviados entre cliente y empresa.
•	Notificaciones: El campo leido en mensajes_soporte es clave para que el equipo de frontend ponga un "punto rojo" o aviso en la interfaz de la PyME cuando tenga preguntas nuevas de turistas.
•	Uso del Chatbot: Podrían usar el id de estos chats para que, si el dueño no responde en 5 minutos, la IA (Gemma 3) intente dar una respuesta automática basada en el contexto_ia y los productos del negocio.

🟢 Nivel 1: Tablas "Cimiento" (Obligatorias)
Sin estas tablas, la API simplemente no puede arrancar porque son el corazón de la navegación y la identidad.
•	perfiles: Esencial para el login y saber quién es quién .
•	negocios y productos: La razón de ser de la app; si estas fallan, el mapa está vacío .
•	colecciones y amenidades: Los filtros que permiten al usuario encontrar lo que busca .
•	pois: Vital para que la IA (Gemma 3) tenga de dónde sacar el contexto_ia .
🟡 Nivel 2: Tablas de "Promesa Técnica" (Diferenciadores)
Estas tablas cumplen lo que prometieron sobre gamificación y rutas optimizadas. Deberían ser su siguiente enfoque tras el mapa.
•	visitas: Es el sensor que detecta que alguien llegó al lugar .
•	insignias y usuario_insignias: Lo que motiva al usuario a seguir usando la app.
•	recorridos y recorrido_nodos: Registran el esfuerzo físico y la ruta calculada por el algoritmo .
🔴 Nivel 3: Tablas de "Escalabilidad" (Las que pueden quedar sin usar)
Estas son las que mencionas. Están perfectamente diseñadas, tienen sus políticas de seguridad (RLS) y su estructura lista, pero no pasa nada si la API no les envía datos todavía.
•	resenas: Si el tiempo aprieta, pueden mostrar los negocios sin estrellas por ahora .
•	chats_b2b y mensajes_b2b: Son el nivel más alto de networking empresarial. Dejarlas creadas en la BD es un "gancho" excelente para la documentación técnica, aunque el chat no esté activo en la interfaz .
•	negocio_stats: Pueden recolectar datos después; lo importante es que la estructura ya existe para el futuro dashboard .
